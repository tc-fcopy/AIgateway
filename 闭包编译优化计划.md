# 闭包编译优化计划 - PipelinePlanMiddleware

## 概述

本计划使用闭包编译方案优化网关执行链性能，在保持动态编排灵活性的同时，将性能提升到接近静态中间件的水平。

## 问题分析

当前实现存在以下性能瓶颈：

1. **每次请求都重新编译 Handler 链**：`Executor.compileHandlers()` 在每次请求时都会调用
2. **重复的插件查找**：从 registry 多次查找相同的插件
3. **重复的 Enabled 检查**：虽然有 Plan，但插件执行时仍需检查
4. **未利用 Plan 缓存**：Plan 已经缓存，但编译好的执行链没有缓存

## 优化目标

- [ ] 减少 30-50% 的请求延迟
- [ ] 减少 30-60% 的内存分配
- [ ] 保持向后兼容，不破坏现有功能
- [ ] 保持 API 不变，对调用方透明

## 执行步骤

### 第一步：扩展 Plan 结构体，添加 CompiledHandler 字段

- 在 `Plan` 结构体中添加 `CompiledHandler gin.HandlerFunc` 字段
- 添加 `CompiledHandlers` 字段用于存储完整的 handlers chain（可选，用于调试）

**文件**：`http_proxy_pipeline/planner.go` 或新建 `compiler.go`

- 在闭包中直接调用插件，跳过 ShouldExecute 检查（因为 Plan 已确定）

## 技术细节

### 闭包编译示例

```go
func compileExecChain(specs []PluginSpec, registry *http_proxy_plugin.Registry, planVersion string) gin.HandlerFunc {
    var handlers []gin.HandlerFunc

    for _, spec := range specs {
        plugin, ok := registry.Get(spec.Name)
        if !ok {
            continue // 或返回错误
        }

        // 预编译单个插件的 handler
        if middlewarePlugin, ok := plugin.(MiddlewarePlugin); ok && middlewarePlugin.Handler() != nil {
            legacy := middlewarePlugin.Handler()
            handlers = append(handlers, func(c *gin.Context) {
                ec := http_proxy_plugin.NewExecContext(c)
                ec.PlanVersion = planVersion
                // 注意：不再需要 plugin.Enabled 检查，因为 Plan 已保证
                legacy(c)
            })
        } else {
            handlers = append(handlers, func(c *gin.Context) {
                ec := http_proxy_plugin.NewExecContext(c)
                ec.PlanVersion = planVersion
                result := plugin.Execute(ec)
                if result.IsAbort() {
                    handlePluginAbort(c, result)
                    return
                }
            })
        }
    }

    // 返回完整的执行链闭包
    return func(c *gin.Context) {
        for _, h := range handlers {
            h(c)
            if c.IsAborted() {
                return
            }
        }
        // 所有插件执行完后调用 c.Next()
        c.Next()
    }
}
```

### 关键优化点

1. **移除运行时 ShouldExecute 检查**：因为 Plan 已经确定了哪些插件需要执行
2. **预获取所有插件**：从 registry 一次性获取，而不是每次请求都获取
3. **闭包捕获插件引用**：避免重复查找
4. **缓存执行链**：存储在 Plan 中，随 Plan 一起缓存

## 风险与降级方案

### 风险

1. 编译执行链可能消耗更多内存
2. 第一次编译可能有轻微延迟
3. 复杂的闭包可能增加调试难度

### 降级方案

1. 保留原有执行逻辑作为 fallback
2. 添加配置开关，可以禁用预编译
3. 添加监控，记录预编译的使用率和性能指标

## 后续优化方向

1. 使用 `sync.Pool` 缓存 `Plan` 对象
2. 预编译最常用的几种 Plan 组合
3. 添加性能指标收集
4. 考虑使用代码生成（方案二）进一步优化
